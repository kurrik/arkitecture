A simple pipeline

Three top-level nodes laid out left to right, joined by arrows. With no anchors given, each arrow auto-routes to the nearest facing edges. A bare margin: 20 at the @layout root is the document-wide default margin — one knob to space the whole diagram out.

pipeline.ark
webapp {
  label: "Web App"
}
api {
  label: "API"
}
database {
  label: "Database"
}

webapp --> api
api --> database

@layout {
  margin: 20
}

Orthogonal routing (route: orthogonal)

Set route: orthogonal once at the @layout root and every arrow is drawn as an axis-aligned path between its facing edges instead of a straight diagonal. Here the CDN sits high on the left and the Database low on the right, so their connector turns through the gap as a clean Z — and it travels in its own lane: the gap between the two contexts widens to hold the line, so it sits in breathing room instead of crowding either box. (The default, route: straight, is the auto-cardinal line above.)

routing.ark
edge {
  label: "Edge"
  cdn { label: "CDN" }
}
origin {
  label: "Origin"
  api      { label: "API Server" }
  database { label: "Database" }
}

@layout {
  route: orthogonal
  margin: 16
  edge   { direction: vertical }
  origin { direction: vertical }
}

edge.cdn --> origin.database

Routing around obstacles

In route: orthogonal mode an arrow routes around the boxes that lie between its endpoints rather than cutting through them. The link from Client to the Database would run straight through Cache, so it detours up and over it as an axis-aligned path — and travels in its own lane: the row's top perimeter widens to hold the line above the boxes. Anchors pinned to an edge leave perpendicular to that edge, so you stay in control of where a line departs.

detour.ark
# With route: orthogonal an arrow routes *around* the boxes between its endpoints
# instead of cutting through them. The link from Client to the Database would run
# straight through Cache, so it detours up and over it as an axis-aligned path.
tier {
  label: "Request path"
  client   { label: "Client" }
  cache    { label: "Cache" }
  database { label: "Database" }
}

@layout {
  route: orthogonal
  margin: 16
  tier { direction: horizontal }
}

tier.client --> tier.database

Lanes for parallel arrows

When several arrows run along the same channel, each gets its own lane: the channel widens to hold them all and the parallel runs sit side by side instead of overlapping. Here both UI → Data and BFF → Auth cross the gap between the columns, so the gap opens into two lanes.

lanes.ark
# Two arrows that both cross the gap between the columns each get their own lane:
# the gap widens to hold both, so the parallel vertical runs sit side by side
# instead of overlapping. web.ui --> core.data drops down one lane; web.bff -->
# core.auth climbs the other.
web {
  label: "Web"
  ui  { label: "UI" }
  bff { label: "BFF" }
}
core {
  label: "Core"
  auth { label: "Auth" }
  data { label: "Data" }
}

@layout {
  route: orthogonal
  margin: 16
  web  { direction: vertical }
  core { direction: vertical }
}

web.ui  --> core.data
web.bff --> core.auth

Nesting & direction

Nodes nest to any depth, and a separate @layout block arranges each parent's children with a direction. Here a vertical Platform stacks a gateway above a horizontal row of services.

nesting.ark
platform {
  label: "Platform"

  gateway {
    label: "API Gateway"
  }

  services {
    label: "Services"

    users   { label: "Users" }
    orders  { label: "Orders" }
    billing { label: "Billing" }
  }
}

@layout {
  platform          { direction: vertical }
  platform.services { direction: horizontal }
}

Bounded contexts with anchors

A Domain-Driven Design sketch: two bounded contexts, a borderless box: none node that groups Cart and Checkout for layout only, and an arrow routed through named anchors and a nested node path. With route: orthogonal, the arrow leaves the Orders bottom anchor straight down, follows the channel below it and around Ordering Context, and enters Inventory Context from the left — each run a full margin clear of the boxes.

contexts.ark
ordering {
  label: "Ordering Context"

  row {
    cart     { label: "Cart" }
    checkout { label: "Checkout" }
  }

  orders {
    label: "Orders"
    anchors: [db]
  }
}

inventory {
  label: "Inventory Context"
  anchors: [api]
}

@layout {
  route: orthogonal
  margin: 16
  ordering     { direction: vertical }
  ordering.row { box: none; margin: 0; direction: horizontal }

  ordering.orders { anchor db: [0.5, 1.0] }
  inventory       { anchor api: [0.0, 0.5] }
}

ordering.orders#db --> inventory#api

Reusable layout (@block & @use)

A named @block bundles layout you import with @use — explicit and opt-in, with no cascade matching nodes for you. Both groups pull in the same borderless row; the second then overrides its direction directly, because a direct property always beats an imported one.

reuse.ark
app {
  label: "App"

  group1 { web { label: "Web" }   api { label: "API" } }
  group2 { jobs { label: "Jobs" } cron { label: "Cron" } }
}

@layout {
  # A reusable layout bundle. There is no cascade: @use is an explicit, opt-in
  # import, and a property set directly on a node overrides the imported one.
  @block row { box: none; direction: horizontal }

  app        { direction: vertical }
  app.group1 { @use row }                       # imported: box none + horizontal
  app.group2 { @use row; direction: vertical }  # same block, direction overridden
}

Semantic kinds (kind)

A node's kind is a semantic tag — what the component is — that implicitly imports the @block of the same name as a layout baseline. Both tiers share one tier rule instead of repeating it, and the built-in invisible kind makes external a borderless grouping.

kinds.ark
platform {
  label: "Platform"

  # `kind` is a semantic tag — what the node *is*. A custom kind imports the
  # @block of the same name as a layout baseline, so both tiers lay out the same
  # way from one rule, with no per-node repetition.
  edge { kind: tier; cdn { label: "CDN" } lb { label: "Load Balancer" } }
  core { kind: tier; app { label: "App" } db { label: "Database" } }

  # The built-in `invisible` kind applies box: none, so this node groups its
  # children for layout only — no border — while keeping its id and path.
  external {
    kind: invisible
    stripe { label: "Stripe" }
    twilio { label: "Twilio" }
  }
}

@layout {
  @block tier { direction: horizontal }

  platform          { direction: vertical }
  platform.external { direction: horizontal }
}

Regrouping with @group

Inside a node's @layout you can list its children to reorder them and wrap a run of them in an anonymous @group for purely visual nesting. Here a vertical group of two services sits, as one unit, to the left of Payments — the group draws no border and adds no path segment.

arrangement.ark
services {
  label: "Services"

  userService  { label: "User Service" }
  orderService { label: "Order Service" }
  payments     { label: "Payments" }
}

@layout {
  services {
    direction: horizontal

    # Wrap two children in an anonymous @group so they stack vertically and sit,
    # as one unit, to the left of payments. A @group is invisible and has no id.
    @group { direction: vertical; userService; orderService }
    payments
  }
}

Styling (colour & stroke width)

Hex colours and stroke widths live in @layout like any other property: borderWidth/borderColor/backgroundColor style a node's box, and pathWidth/pathColor style the arrows that start at it. Each is settable per node, document-wide (a bare property at the sheet root — the fallback here paints every box), or bundled in a @block (the blue border on Web Tier is one @use accent). There is no cascade: a direct value wins, and a child never inherits a parent's colour.

styling.ark
web {
  label: "Web Tier"
  ui  { label: "UI" }
  api { label: "API" }
}
cache { label: "Cache" }
db    { label: "Database" }

web.api --> cache
web.api --> db

@layout {
  # A bare style property at the sheet root is the DOCUMENT-WIDE default — the
  # fallback for every node and arrow that sets none. Colours are hex.
  borderColor: #334155
  backgroundColor: #f8fafc

  # Bundle a look into a @block and pull it in with @use, like any other layout.
  @block accent { borderColor: #2563eb; borderWidth: 2 }
  web { direction: horizontal; @use accent }   # blue 2px border, imported

  # pathColor / pathWidth style the arrows that START at this node.
  web.api { pathColor: #2563eb; pathWidth: 2 }

  cache { backgroundColor: #fef9c3 }   # per-node fill override of the default
  db    { borderColor: #16a34a; borderWidth: 3 }
}
